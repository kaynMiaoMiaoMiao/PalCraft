# PalCraft 开发计划书

版本：0.1  
目标 Minecraft 版本：1.20.1  
目标 Mod Loader：Fabric  
当前项目状态：基础 Fabric 开发环境已建立，已有 `palcraft:capture_orb` 物品、自定义创造模式物品栏和基础资源结构。

## 1. 项目定位

PalCraft 的目标是制作一个 Minecraft 生存向的原创伙伴捕捉、养成、战斗和据点自动化 Mod。

玩法方向可以参考“捕捉伙伴、召唤伙伴、伙伴协助战斗、伙伴在据点工作、逐步扩展生产链”的大框架，但项目需要保持原创实现：

- 不使用其他商业游戏的名称、角色、美术、音效、UI、文案、图标和专有设定。
- 伙伴生物、技能、道具、据点设施都使用原创命名和原创美术。
- 可以借鉴通用玩法类型，例如捕捉、养成、自动化、骑乘、Boss、科技树，但具体表达必须重新设计。

## 2. 最小可玩版本目标

第一个可玩版本先做小，不追求内容数量，优先把核心闭环跑通。

MVP 范围：

- 一个可投掷的捕捉球。
- 至少一个可被捕捉的原创伙伴生物。
- 捕捉成功后，伙伴进入玩家的伙伴数据仓库。
- 玩家可以召唤、收回当前伙伴。
- 伙伴可以跟随玩家，并攻击玩家正在战斗的目标。
- 一个基础伙伴属性结构：等级、经验、生命、攻击、元素类型。
- 一个专属创造模式物品栏，集中放置本 Mod 物品。
- `./gradlew build` 稳定通过。

MVP 不包含：

- 大量伙伴种类。
- 完整据点自动化。
- 复杂 UI。
- 繁殖、词条、科技树、枪械、飞行骑乘。
- 服务端大规模性能优化。

## 3. 技术路线

基础技术：

- Fabric API：注册物品、实体、方块、事件、网络同步。
- Java 17：Minecraft 1.20.1 的目标 Java 版本。
- Gradle Loom：构建、运行客户端、打包。

建议逐步引入的依赖：

- GeckoLib：当开始制作复杂伙伴模型和动画时引入。
- ModMenu + Cloth Config：当需要配置界面时引入。
- REI 或 EMI 兼容：当配方和生产链变复杂后再考虑。

代码包建议：

```text
com.bmht.palcraft
com.bmht.palcraft.registry       注册入口：物品、方块、实体、物品栏
com.bmht.palcraft.item           道具逻辑：捕捉球、召唤道具
com.bmht.palcraft.entity         伙伴实体和投掷物实体
com.bmht.palcraft.capture        捕捉概率、捕捉流程
com.bmht.palcraft.partner        伙伴实例数据、仓库、召唤逻辑
com.bmht.palcraft.network        客户端/服务端网络包
com.bmht.palcraft.base           据点核心、工作调度
com.bmht.palcraft.client         客户端渲染、模型、UI
```

资源目录建议：

```text

src/main/resources/assets/palcraft/textures/item
src/main/resources/assets/palcraft/models/entity
src/main/resources/assets/palcraft/textures/entity
src/main/resources/assets/palcraft/lang

```

## 4. 核心数据设计

伙伴实例数据建议不要只存在于实体 NBT。捕捉后应保存为玩家数据，召唤时再生成实体。

建议结构：

```text
PalInstance
- instanceUuid
- speciesId
- customName
- ownerUuid
- level
- exp
- health
- maxHealth
- attack
- defense
- elementType
- skills
- traits
- capturedAt
```

玩家伙伴仓库：

```text
PlayerPalData
- ownerUuid
- activeSlot
- partySlots
- storedPals
```

后续据点数据：

```text
BaseData
- baseUuid
- ownerUuid
- dimension
- coreBlockPos
- radius
- assignedPals
- workQueue
- storageLinks
```

数据保存候选方案：

- 前期可以先使用实体 NBT 和玩家相关 PersistentState 快速实现。
- 数据复杂后再抽象成单独的 `partner` 数据层。
- 多人服务器中，关键逻辑必须以服务端为准，客户端只负责显示和请求。

## 5. 阶段开发计划

### 阶段 0：项目骨架

状态：已基本完成。

已完成内容：

- Fabric 1.20.1 项目环境。
- Gradle Wrapper。
- 基础 Mod 入口。
- 自定义创造模式物品栏。
- 捕捉球物品和贴图。
- `./gradlew build` 构建成功。

验收标准：

- 游戏能加载 Mod。
- 创造模式中能看到 PalCraft 分类栏。
- 分类栏内能看到捕捉球。

### 阶段 1：注册结构整理

状态：已完成。

目标：把当前集中在 `PalCraft.java` 的注册逻辑拆分，为后续扩展做准备。

任务：

- 新建 `ModItems`。
- 新建 `ModItemGroups`。
- 保留 `PalCraft` 作为总初始化入口。
- 给捕捉球建立独立 `CaptureOrbItem` 类，为右键投掷做准备。

验收标准：

- `./gradlew build` 通过。
- 创造模式物品栏不丢失。
- 捕捉球物品 ID 不变：`palcraft:capture_orb`。

### 阶段 2：捕捉球投掷物

状态：已完成代码实现，`./gradlew build` 已通过；游戏内投掷效果待 `runClient` 手动复核。

目标：捕捉球可以右键投掷，生成服务端投掷物实体。

任务：

- 注册 `CaptureOrbEntity`。
- `CaptureOrbItem#use` 生成投掷物。
- 添加基础飞行轨迹、重力和碰撞检测。
- 添加投掷音效和基础粒子，先可用原版资源。

验收标准：

- 生存模式右键会消耗一个捕捉球。
- 创造模式右键不消耗。
- 捕捉球能飞出并碰撞方块或实体。
- 多人环境下实体由服务端生成。

### 阶段 3：基础捕捉流程

状态：已完成代码实现，`./gradlew build` 已通过；捕捉成功/失败需要阶段 4 的可捕捉伙伴实体联调。

目标：捕捉球命中指定生物后，能进行捕捉判定。

任务：

- 新建 `CaptureService`。
- 定义可捕捉目标规则。
- 初版只允许捕捉 PalCraft 自己注册的伙伴生物。
- 实现捕捉概率公式。
- 捕捉成功后移除目标实体，并生成 `PalInstance`。
- 捕捉失败时播放失败反馈。

初版捕捉概率建议：

```text
baseChance = speciesBaseChance
healthFactor = 1.0 - currentHealth / maxHealth
levelPenalty = targetLevel > playerProgress ? reducedChance : 0
finalChance = clamp(baseChance + healthFactor * 0.45 - levelPenalty, 0.05, 0.95)
```

验收标准：

- 命中可捕捉生物时触发捕捉判定。
- 成功和失败都有明确反馈。
- 捕捉成功不会复制实体或丢数据。

### 阶段 4：第一个原创伙伴生物

状态：已完成代码实现，`./gradlew build` 已通过；游戏内自然生成、创造蛋和捕捉流程待 `runClient` 手动复核。

目标：实现一个最小可用伙伴生物，作为所有系统的测试对象。

建议第一个生物：

- 临时代号：`sparkit`
- 定位：小型电属性伙伴。
- 行为：自然生成、被攻击会反击、可被捕捉。
- 美术：前期可以先用简单模型或临时贴图，后续用 Blockbench/GeckoLib 替换。

任务：

- 注册伙伴实体类型。
- 添加基础属性。
- 添加自然生成规则，或先只通过创造蛋生成。
- 添加掉落表。
- 添加语言文件。

验收标准：

- 能在游戏中生成测试伙伴。
- 捕捉球能对它进行捕捉流程。
- 捕捉成功后能保存基本物种 ID 和属性。

### 阶段 5：伙伴仓库与召唤

状态：已完成并通过游戏内验收。

目标：捕捉成功后的伙伴不只是消失，而是进入玩家可管理的数据仓库。

任务：

- 实现 `PlayerPalData`。
- 实现出战槽位，初期只做 1 个槽。
- 添加召唤和收回逻辑。
- 可以先用命令测试，再做正式 UI。

建议调试命令：

```text
/palcraft list
/palcraft summon 0
/palcraft recall
```

验收标准：

- 捕捉成功后 `/palcraft list` 能看到伙伴。
- `/palcraft summon 0` 能召唤伙伴实体。
- `/palcraft recall` 能收回伙伴。
- 退出重进后伙伴数据仍存在。

### 阶段 6：伙伴 AI 战斗闭环

状态：已完成并通过游戏内验收。

目标：召唤出的伙伴能作为玩家助手参与战斗。

任务：

- 跟随主人。
- 距离过远时传送到主人附近。
- 主人攻击目标时，伙伴协助攻击。
- 主人被攻击时，伙伴反击攻击者。
- 伙伴低血量时停止冒进或回到主人身边。

验收标准：

- 伙伴不会频繁卡死或乱跑。
- 伙伴不会攻击主人。
- 伙伴死亡或收回后数据状态正确。

### 阶段 7：属性、经验和技能

状态：已完成并通过游戏内验收。

目标：让伙伴拥有成长空间。

任务：

- 等级和经验。
- 基础属性成长。
- 技能数据结构。
- 先实现 3 个技能：
  - 近战撞击。
  - 远程能量弹。
  - 自我恢复或护盾。

验收标准：

- 战斗后伙伴获得经验。
- 升级后属性提高。
- 技能释放有冷却，不会每 tick 触发。

### 阶段 8：据点核心 MVP

状态：已完成并通过游戏内验收。

目标：进入“伙伴工作”玩法，但先只做最小调度。

任务：

- 注册 `base_core` 方块。
- 放置后创建据点范围。
- 允许把伙伴分配到据点。
- 据点先支持一个工作类型：搬运或采集。

验收标准：

- 玩家能放置据点核心。
- 据点能保存到世界数据。
- 分配伙伴后，伙伴会在据点范围内执行简单任务。

### 阶段 9：据点工作调度

状态：已完成并通过游戏内验收；已补充采矿、伐木、成熟作物收获并补种的实际世界交互。

目标：从单个工作扩展到可维护的任务系统。

任务：

- 工作类型接口：采矿、伐木、种植、搬运、制造。
- 任务队列。
- 搜索限频，避免性能问题。
- 伙伴工作适性。

验收标准：

- 多个伙伴能在同一据点分工。
- 不会每 tick 大范围扫描方块。
- 区块卸载和重载后状态可恢复。

### 阶段 10：UI 与体验

状态：已完成并通过游戏内验收；已补充现代化界面样式、汉化、据点核心右键打开、伙伴改名和服务端同步操作。

目标：把命令调试流程替换成玩家可理解的界面。

任务：

- 伙伴列表界面。
- 出战槽位界面。
- 据点分配界面。
- 伙伴详情面板。

验收标准：

- 不依赖命令也能完成召唤、收回、查看伙伴。
- 客户端 UI 数据由服务端同步。

### 阶段 11：内容扩展

目标：在核心系统稳定后增加内容。

建议第一批内容：

- 3 到 5 个原创伙伴。
- 3 种元素类型。
- 6 到 10 个技能。
- 2 到 3 种捕捉球等级。
- 3 个基础据点方块。

据点工作站规划：

- 新增一组原创据点工作站方块，工作站只提供生产入口，不直接存储大量产物。
- 第一批工作站建议为：矿物加工台、木材加工台、石材加工台、食物料理台、通用装配台。
- 工作站需要被据点扫描识别，只有处于据点范围内并归属同一据点时才可分配宠物。
- 宠物分配到工作站后生成生产任务，任务完成后产物优先进入据点范围内的箱子、木桶等存储方块。
- 每个工作站先提供少量固定配方，例如矿物加工台产出基础矿石或矿物碎片，木材加工台产出原木或木板，石材加工台产出石材，食物料理台产出基础食物。
- 制造类任务不再使用虚拟库存计数；所有产物都应该以真实物品形式落入据点存储。
- 后续再扩展为配方数据驱动，避免把生产表硬编码在工作调度里。

验收标准：

- 新增内容通过数据和注册表扩展，不破坏核心系统。
- 每个伙伴至少有一个明确玩法定位。

## 6. 开发顺序原则

每个阶段都按这个顺序推进：

```text
设计本阶段最小范围
实现服务端核心逻辑
补客户端显示资源
运行 ./gradlew build
运行 runClient 手动验证
修复问题
记录结果
进入下一阶段
```

不要提前堆大量美术和内容。核心逻辑不稳定时，内容越多，后续返工越重。

## 7. 验收和测试清单

每个功能合并前至少检查：

- `./gradlew build` 成功。
- 游戏能启动到主菜单。
- 新增注册项没有 missing model 或 missing texture。
- 单人世界中功能可用。
- 如果涉及网络包，局域网或 dedicated server 至少测一次。
- 如果涉及存档数据，退出重进后数据不丢。
- 如果涉及实体 AI，观察是否有明显 TPS 下降。

## 8. 主要风险

技术风险：

- 实体 AI 和寻路容易卡顿。
- 据点自动化如果扫描过频，会影响 TPS。
- 多人同步如果设计不清晰，容易出现客户端显示和服务端状态不一致。
- 存档数据结构如果频繁变化，后续需要迁移逻辑。

内容风险：

- 伙伴数量过早扩张会拖慢核心系统开发。
- 美术、动画、音效工作量很大，应该逐步替换临时资源。

合规风险：

- 不能使用其他游戏的角色、名称、图标、模型、贴图、音效或 UI。
- 不能把项目宣传成某个商业游戏的复制版或移植版。

## 9. 当前下一步任务

建议下一轮先复核“阶段 10：UI 与体验”的优化项。

具体任务：

1. 运行 `./gradlew runClient`，进入测试世界。
2. 确保已有至少一个捕获伙伴和一个据点核心。
3. 按 `P` 打开随身宠物界面，右键据点核心打开据点管理界面，确认界面样式和中文显示正常。
4. 在界面中选择伙伴，点击“召唤”和“收回”，确认服务端状态同步。
5. 在名字输入框中修改伙伴名字并保存，确认列表、详情和出战实体名称更新。
6. 捕获超过 4 只伙伴，确认新增伙伴进入据点宠物仓库。
7. 在据点管理的“工作分配”页签中分配仓库宠物，确认宠物实体出现在据点附近。
8. 在据点范围内放置箱子或木桶，再放置可采矿方块、原木和成熟作物，等待任务完成后确认产物进入存储方块。
